### 開発・実装内容
#### 1. タスク管理用カレンダーの実装
【概要】  
タスク管理用カレンダーの実装を担当した。

【どのような機能の開発・実装か】  
- システムに登録されている未完了のタスクを担当者ごとに取得し、カレンダーに表示する機能
  - 登録されているタスクの取得用APIの実装
  - JSライブラリのFullCalendarを使用したカレンダーUIの実装

【課題・問題点】
1. 他プロジェクトでFullCalendarライブラリをJavaScriptで直接DOMを操作する形で実装していた例が存在していた。参考にしようとしたが、状態管理を手動で行う必要があるため、コードが複雑になり保守性が低いと感じていた
2. タスクのデータ総量が膨大なため、データ取得時の検索クエリが重くなってしまっていた

【打ち手・使用した技術】  
1. FullCalendarをVue版で使用し、コンポーネントベースの開発、宣言的なUIと状態管理により保守性の高い実装を行なった
2. インデックスを追加し、取得処理のパフォーマンスチューニングを行なった

#### 2. システムにアップロードしたファイルのFAX送信機能の実装
【概要】  
システムにアップロードされたファイルを、FAX送信できる機能の実装を担当した。

【どのような機能の開発・実装か】
他社製のメールFAXサービスとAWS SESを使用したFAX送信機能
エンジニア3名で対応し、
1. システムとメールFAXサービスの連携機能
2. 送信履歴の登録
3. 選択したファイルの結合とpdf変換
4. 送信画面のUI

上記のうち1, 2を担当しました。

【課題・問題点】
1. 業務フローが複雑だったため、考慮もれやエンジニア間での認識の相違が発生するリスクがあった
2. 3名で1つの機能を実装したことがなかったので、実装内容をマージした時に大量のコンフリクトや想定通りに動作しないリスクがあった
3. 何も考えずに実装するとシステムが他社製のメールFAXサービスに対して密結合となってしまうリスクがあった

【打ち手・使用した技術】
1. Figjamでイベントストーミングによるフロー図の作成や、ロバストネス図の作成を行なった。これを用いて実装着手前にエンジニア間で業務フローと実装方針を確認し、見落としていたドメイン知識の補完を行なった。
2. Gitのブランチ戦略を工夫した。mainからマージ用のブランチを作成し、そこから各自のブランチを作成するようにした。毎日Gitlabで各自の作業内容のMRを作成し、マージ用のブランチにマージした。それを各自の作業ブランチに取り込みを行うことでコンフリクトの早期解消や他メンバーの実装内容の把握を行なった。
3. ドメイン層のアップロードファイルに関わるディレクトリにFAX送信用のインターフェースを配置し、具体的な送信処理はインフラストラクチャ層に配置することで密結合になることを防いだ。
